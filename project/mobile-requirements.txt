Mobile App Requirements for Aquila System

1. Загальна логіка
-----------------
- Додаток має дві ролі: Адміністратор і Користувач.
- При старті додатку пропонується вибір:
  - Я Адмін
  - Я Користувач
- Для адміністратора використовуються дефолтні дані:
  - логін: admin
  - пароль: admin
- Користувач може зайти тільки через акаунт, який створив адміністратор.
- Адміністратор керує кімнатами, пристроями та звичайними користувачами.
- Користувач бачить тільки інфу: вимірювання, повідомлення, снімки, стан пристроїв.
- Потрібна підтримка push-повідомлень для алертів / важливих повідомлень.

2. Ролі
-------
Адміністратор:
- Створює/редагує/видаляє кімнати.
- Створює/редагує/видаляє пристрої.
- Створює/редагує/видаляє звичайних користувачів.
- Переглядає всі алерти та може їх помічати прочитаними/непрочитаними.
- Може переглядати історію вимірювань по пристроях та кімнатах.
- Може переглядати снімки пристроїв.
- Має доступ до списку всіх користувачів.

Користувач:
- Переглядає список кімнат і пристроїв.
- Переглядає історію вимірювань по пристрою.
- Переглядає повідомлення/алерти, які стосуються пристроїв.
- Отримує push-повідомлення про важливі тривоги.
- Переглядає снімки пристроїв.

3. Структура мобільного додатку
-------------------------------
3.1 Екран вибору ролі
- Кнопки: "Я Адмін" і "Я Користувач".
- Переходить до екрану логіну.

3.2 Login екран
- Поля: Email/Login, Password.
- Для адміністратора можна показати hint: admin / admin.
- Після успішного входу зберігати роль і токен/сесію.

3.3 Головний екран Адміна
- Карти/кнопки:
  - Кімнати
  - Пристрої
  - Користувачі
  - Алерти
  - Снімки
  - Статистика / Останні вимірювання
- Можливість logout.

3.4 Головний екран Користувача
- Карти/кнопки:
  - Кімнати
  - Пристрої
  - Вимірювання
  - Повідомлення / Алерти
  - Снімки
- Можливість logout.

3.5 Кімнати
- Список кімнат з кількістю пристроїв.
- Для адміна: кнопки "Створити кімнату", "Редагувати", "Видалити".
- Для обох: вибір кімнати відкриває деталі кімнати і список пристроїв у кімнаті.
- Деталі кімнати повинні містити назву, тип, опис, список пристроїв.

3.6 Пристрої
- Список пристроїв для адміністратора і користувача.
- Кожен пристрій показує:
  - назва,
  - MAC,
  - тип,
  - статус активності,
  - кімната.
- Для адміна: кнопки "Додати пристрій", "Редагувати", "Видалити".
- При виборі пристрою показати подробиці і доступ до історії телеметрії.

3.7 Користувачі (для адміністратора)
- Список звичайних користувачів.
- Кнопка "Створити користувача".
- Деталі користувача: email, повне ім’я, роль.
- Редагувати/видаляти користувача.

3.8 Повідомлення / Алерти
- Список алертів (усі або тільки непрочитані).
- Кнопка "Позначити як прочитаний" / "Позначити як непрочитаний".
- Для адміна: всі алерти системи.
- Для користувача: алерти, які стосуються його пристроїв або кімнат (залежно від реалізації).
- Повідомлення повинні мати тип, текст, час, статус прочитано/непрочитано.

3.9 Історія вимірювань
- Показувати дані за пристроєм:
  - температура,
  - вологість,
  - освітленість,
  - рівень шуму.
- Рекомендовано графіки/тренди.
- Підтримка параметрів limit/offset для пагінації.

3.10 Снімки
- Список снімків для пристрою.
- Кожен снімок має URL, час, причину.
- Для адміна: можна видаляти снімок.

4. API endpoint-и для мобільного клієнта
-----------------------------------------
4.1 Аутентифікація / користувачі
- POST /api/users/register
- POST /api/users/login
- GET /api/users
- GET /api/users/:id
- PUT /api/users/:id
- PATCH /api/users/:id/password
- DELETE /api/users/:id

4.2 Кімнати
- GET /api/rooms
- POST /api/rooms
- GET /api/rooms/:id
- GET /api/rooms/:roomId/telemetry
- PUT /api/rooms/:id
- DELETE /api/rooms/:id

4.3 Пристрої
- GET /api/devices
- POST /api/devices
- GET /api/devices/by-mac/:macAddress
- GET /api/devices/:id
- GET /api/devices/:deviceId/telemetry
- GET /api/devices/:deviceId/alerts
- PUT /api/devices/:id
- DELETE /api/devices/:id

4.4 Телеметрія
- POST /api/telemetry
- GET /api/telemetry/:deviceId

4.5 Снімки
- POST /api/snapshots
- GET /api/snapshots/:deviceId
- DELETE /api/snapshots/:id

4.6 Алерти
- POST /api/alerts
- GET /api/alerts
- GET /api/alerts/unread
- GET /api/alerts/unread/count
- GET /api/alerts/:deviceId
- PUT /api/alerts/:id/read
- PUT /api/alerts/:id/unread
- DELETE /api/alerts/:id

5. Умови використання ролей і логінів
-------------------------------------
- Адміністратор:
  - логін: admin
  - пароль: admin
  - може керувати усім.
- Користувач:
  - повинен бути створений адміністратором через CRUD Admin UI.
  - може лише переглядати дані і отримувати повідомлення.

6. Екрани мобайл додатку
------------------------
6.1 Вибір ролі
- "Я Адмін" / "Я Користувач"
- Переходить на екран входу.

6.2 Екран входу
- Поля: логін/email, пароль.
- Для адміна використовувати admin/admin.
- Для користувача входити через створений обліковий запис.

6.3 Дашборд адміністратора
- Кнопки:
  - Кімнати;
  - Пристрої;
  - Користувачі;
  - Алерти;
  - Снімки;
  - Статистика / останні дані.

6.4 Дашборд користувача
- Кнопки:
  - Кімнати;
  - Пристрої;
  - Вимірювання;
  - Повідомлення;
  - Снімки.

6.5 Кімнати
- Список кімнат з пристроями.
- Для адміна: CRUD кімнати.
- Для користувача: перегляд.

6.6 Пристрої
- Список пристроїв.
- Для адміна: CRUD пристрої.
- Для користувача: перегляд і доступ до телеметрії.

6.7 Користувачі
- Для адміна: створення, редагування, видалення звичайних користувачів.
- Для користувача: немає доступу.

6.8 Повідомлення / Алерти
- Список повідомлень / алертів.
- Для адміна: управління статусом прочитаності.
- Для користувача: перегляд і push-повідомлення.

6.9 Вимірювання
- Перегляд історії сенсорних даних пристрою.
- Можливість пагінації.

6.10 Снімки
- Перегляд снімків із пристрою.
- Для адміна: видалення снімків.

7. Додаткові вимоги
-------------------
- Додаток має працювати у режимі online.
- Потрібне коректне повідомлення про помилки від сервера.
- Для мобільних push-повідомлень можна використовувати Firebase Cloud Messaging або інший сервіс.
- У разі перевищення порогів датчиків показувати попередження.
- Якщо бекенд змінив IP/порт, потрібно коректно змінювати URL бази.

8. Пояснення для розробника мобайлу
-----------------------------------
- Всі API шляхи починаються з /api.
- Якщо користувач залогінений як адміністратор, відкривається admin flow.
- Якщо користувач залогінений як звичайний, відкривається readonly flow.
- Адаптувати інтерфейс залежно від ролі.
- Не слід показувати адміністраторські дії звичайному користувачу.
